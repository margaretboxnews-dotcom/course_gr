# День 2 — Получение эмбеддингов (Hidden States)

## Что такое hidden states?

В День 1 мы научились превращать текст в токены (числа). Но сами по себе эти числа —
просто индексы в словаре, они ничего не "понимают". Настоящая магия происходит,
когда мы пропускаем токены через модель трансформера.

На каждом слое трансформера каждый токен получает новое числовое представление —
вектор из 768 чисел (для DistilBERT). Эти промежуточные представления называются
**hidden states** (скрытые состояния).

```
Вход: input_ids [101, 10747, 18379, 10134, 28149, 19308, 106, 102]
                            ↓
        [Embedding Layer]  →  матрица [8 × 768] (начальные эмбеддинги)
                            ↓
        [Слой 1]           →  матрица [8 × 768] (hidden state после слоя 1)
        [Слой 2]           →  матрица [8 × 768] (hidden state после слоя 2)
        [Слой 3]           →  матрица [8 × 768] (hidden state после слоя 3)
        [Слой 4]           →  матрица [8 × 768] (hidden state после слоя 4)
        [Слой 5]           →  матрица [8 × 768] (hidden state после слоя 5)
        [Слой 6]           →  матрица [8 × 768] (hidden state после слоя 6)
                            ↓
        Выход: last_hidden_state — матрица [8 × 768]
```

**last_hidden_state** — это выход последнего слоя. Именно эти векторы содержат
максимально обогащённые контекстом представления каждого токена.

### Что меняется от слоя к слою?

На начальных слоях представления токенов близки к их словарным значениям.
С каждым слоем self-attention добавляет всё больше контекстной информации:

- **Слои 1–2**: базовые синтаксические паттерны (часть речи, структура предложения)
- **Слои 3–4**: более сложные грамматические отношения (согласование, зависимости)
- **Слои 5–6**: высокоуровневая семантика (смысл, тональность, тема)

Исследования (Jawahar et al., 2019) показали, что нижние слои BERT улавливают
синтаксис, а верхние — семантику. Это похоже на CNN для изображений, где первые
слои находят линии и углы, а последние — объекты целиком.

## CLS-токен как представление предложения

Вспомним из Дня 1: токен `[CLS]` ставится в начало каждого текста. После прохождения
через все слои трансформера его вектор аккумулирует информацию обо всём предложении
через механизм self-attention.

```python
cls_embedding = outputs.last_hidden_state[:, 0, :]  # shape: [batch_size, 768]
```

Разберём индексацию:
- `[:, 0, :]` — берём **все** примеры в батче (`:`)
- позицию **0** (первый токен = `[CLS]`)
- **все** 768 измерений (`:`)

Этот вектор из 768 чисел — компактное числовое представление смысла всего текста.
Его можно использовать для:
- Классификации (подать на линейный слой → класс)
- Сравнения текстов (косинусное сходство между CLS-векторами)
- Кластеризации (сгруппировать похожие тексты)
- Визуализации (снизить размерность до 2D через t-SNE или UMAP)

### Альтернативы CLS-токену

CLS-токен — не единственный способ получить представление предложения:

| Метод | Формула | Когда лучше |
|-------|---------|-------------|
| CLS-токен | `hidden[:, 0, :]` | Для дообученных моделей (fine-tuned) |
| Mean pooling | `mean(hidden[:, 1:-1, :])` | Для предобученных моделей без fine-tuning |
| Max pooling | `max(hidden[:, 1:-1, :])` | Когда важны ключевые признаки |

**Mean pooling** — усреднение векторов всех токенов (кроме [CLS] и [SEP]) — часто
работает лучше для предобученных моделей, потому что CLS-токен в BERT обучался
специально для задачи NSP, а не для представления предложений.

Модели **Sentence-BERT** (SBERT) специально обучены так, чтобы CLS-токен давал
хорошее представление предложения. Но для обычного BERT/DistilBERT mean pooling
может быть лучше.

## Режим model.eval() и torch.no_grad()

### model.eval()

Переводит модель в **режим оценки** (evaluation mode). Это влияет на два типа слоёв:

- **Dropout** — при обучении случайно "выключает" часть нейронов (для регуляризации).
  В eval-режиме dropout отключается, и все нейроны работают.
- **BatchNorm** — при обучении вычисляет статистики (среднее, дисперсию) на текущем
  батче. В eval-режиме использует накопленные статистики.

Если забыть `model.eval()`, результаты будут нестабильными — каждый запуск
на одних и тех же данных может давать разные числа.

### torch.no_grad()

```python
with torch.no_grad():
    outputs = model(**tokens)
```

Контекстный менеджер, который отключает **вычисление градиентов**. Зачем?

При обучении PyTorch строит **вычислительный граф** — запоминает все операции,
чтобы потом вычислить градиенты (для обратного распространения ошибки). Это:
- Занимает **много памяти** (хранение промежуточных значений)
- Требует **дополнительных вычислений**

Когда мы только получаем предсказания (inference), градиенты не нужны.
`torch.no_grad()` экономит ~30–50% памяти и ускоряет вычисления.

## Разбор outputs модели

```python
outputs = model(**tokens)
```

Оператор `**` распаковывает словарь `tokens` в именованные аргументы:
```python
# Эквивалентно:
outputs = model(
    input_ids=tokens["input_ids"],
    attention_mask=tokens["attention_mask"]
)
```

Объект `outputs` — это `BaseModelOutput` с полями:
- `outputs.last_hidden_state` — тензор `[batch_size, seq_len, 768]` — выход последнего слоя
- `outputs.hidden_states` — (если включено) кортеж из 7 тензоров: эмбеддинги + 6 слоёв
- `outputs.attentions` — (если включено) attention weights каждого слоя

### Форма last_hidden_state

```
outputs.last_hidden_state.shape = torch.Size([1, 8, 768])
                                              │  │   │
                                              │  │   └─ hidden_size: 768 чисел на токен
                                              │  └─ sequence_length: 8 токенов
                                              └─ batch_size: 1 текст
```

Каждый из 8 токенов получил свой вектор из 768 чисел. Эти векторы — контекстуальные:
слово "bank" в "river bank" и "bank account" будет иметь **разные** векторы,
потому что self-attention учитывает соседние слова.

## Функция get_embeddings: зачем батчи?

```python
def get_embeddings(texts, tokenizer, model, batch_size=32):
    for i in range(0, len(texts), batch_size):
        batch_texts = texts[i:i+batch_size]
        ...
```

Почему нельзя подать все тексты сразу? Потому что GPU-память ограничена.
Если у нас 10 000 текстов по 128 токенов, нужно:

```
10000 × 128 × 768 × 4 bytes (float32) = 3.9 GB только для hidden states
+ промежуточные вычисления в 6 слоях ≈ 15–20 GB
```

Разбивая на батчи по 32, мы используем:
```
32 × 128 × 768 × 4 bytes = 12.6 MB — помещается в любую GPU
```

### .cpu().numpy()

```python
cls_embeddings = outputs.last_hidden_state[:, 0, :]
all_embeddings.append(cls_embeddings.cpu().numpy())
```

- `.cpu()` — переносит тензор с GPU на CPU (если модель на GPU). На CPU работает
  NumPy, scikit-learn и другие библиотеки.
- `.numpy()` — конвертирует PyTorch-тензор в NumPy-массив.

### np.vstack

```python
return np.vstack(all_embeddings)
```

Вертикально склеивает массивы из разных батчей:
```
Батч 1: [32 × 768]
Батч 2: [32 × 768]
Батч 3: [32 × 768]
        ↓ np.vstack
Результат: [96 × 768]
```

## Косинусное сходство

### Зачем нужно сравнивать тексты числами?

Когда тексты превращены в векторы, мы можем математически измерить,
насколько они похожи. Это фундамент для:
- Поиска похожих документов
- Рекомендательных систем
- Определения перефразирования (парафраз)
- Кластеризации текстов по темам

### Что такое косинусное сходство?

Это мера сходства двух векторов, основанная на **угле** между ними:

```
cos_sim(A, B) = (A · B) / (||A|| × ||B||)
```

где:
- `A · B` — скалярное произведение (поэлементное умножение + сумма)
- `||A||` — длина (норма) вектора A
- `||B||` — длина (норма) вектора B

Значения:
- `1.0` — векторы указывают в одном направлении (тексты максимально похожи)
- `0.0` — векторы перпендикулярны (тексты не связаны)
- `-1.0` — векторы противоположны (редко встречается в эмбеддингах)

### Почему не евклидово расстояние?

Евклидово расстояние (`||A - B||`) зависит от **длины** векторов.
Два текста могут иметь похожий смысл, но разную "интенсивность" (разную норму вектора).
Косинусное сходство нормализует по длине и смотрит только на **направление**.

Пример:
```
A = [1, 2, 3]      (короткий вектор)
B = [2, 4, 6]      (длинный вектор, то же направление)
C = [3, 1, 0]      (другое направление)

cos_sim(A, B) = 1.0   — то же направление
cos_sim(A, C) = 0.36  — другое направление

euclidean(A, B) = 3.7  — кажутся далёкими!
euclidean(A, C) = 3.7  — кажутся одинаково далёкими!
```

### Ожидаемые результаты

```python
similarity("Great movie!", "Amazing film!")   # ~0.85–0.95
similarity("Great movie!", "Terrible film!")  # ~0.70–0.80
```

Почему "Terrible film" всё ещё имеет высокое сходство с "Great movie"?
Потому что **оба текста о кино**. Модель без fine-tuning улавливает **тему**
(оба про фильмы), но плохо различает **тональность** (positive vs negative).

Для точного разделения по тональности нужно дообучение (fine-tuning) — День 4+.

## AutoModel vs AutoModelForSequenceClassification

В библиотеке Hugging Face есть разные "обёртки" для одной и той же модели:

| Класс | Выход | Применение |
|-------|-------|------------|
| `AutoModel` | last_hidden_state [batch, seq, 768] | Извлечение эмбеддингов, кастомные задачи |
| `AutoModelForSequenceClassification` | logits [batch, num_classes] | Классификация текстов |
| `AutoModelForTokenClassification` | logits [batch, seq, num_classes] | NER, POS-tagging |
| `AutoModelForQuestionAnswering` | start_logits, end_logits | Ответы на вопросы |
| `AutoModelForMaskedLM` | logits [batch, seq, vocab_size] | Заполнение пропусков |

Все они внутри используют один и тот же трансформер (DistilBERT), но добавляют
разные "головы" (heads) — дополнительные слои поверх hidden states.

Сегодня мы используем `AutoModel` — голую модель без головы, чтобы получить
сырые эмбеддинги. В День 4 переключимся на `AutoModelForSequenceClassification`.

## Архитектура DistilBERT изнутри

Когда мы делаем `print(model)`, видим:

```
DistilBertModel(
  (embeddings): Embeddings(
    (word_embeddings): Embedding(119547, 768)     ← таблица токен-эмбеддингов
    (position_embeddings): Embedding(512, 768)     ← таблица позиционных эмбеддингов
    (LayerNorm): LayerNorm((768,))                 ← нормализация
    (dropout): Dropout(p=0.1)                      ← регуляризация
  )
  (transformer): Transformer(
    (layer): ModuleList(
      (0-5): 6 x TransformerBlock(                 ← 6 одинаковых слоёв
        (attention): MultiHeadSelfAttention(
          (q_lin): Linear(768, 768)                ← матрица Wq
          (k_lin): Linear(768, 768)                ← матрица Wk
          (v_lin): Linear(768, 768)                ← матрица Wv
          (out_lin): Linear(768, 768)              ← проекция после конкатенации голов
        )
        (sa_layer_norm): LayerNorm((768,))
        (ffn): FFN(
          (lin1): Linear(768, 3072)                ← расширение в 4 раза
          (lin2): Linear(3072, 768)                ← сжатие обратно
        )
        (output_layer_norm): LayerNorm((768,))
      )
    )
  )
)
```

### Подсчёт параметров

- `word_embeddings`: 119547 × 768 = 91.8M параметров
- `position_embeddings`: 512 × 768 = 393K
- Каждый TransformerBlock:
  - Q, K, V линейные слои: 3 × (768 × 768 + 768) = 1.77M
  - Output projection: 768 × 768 + 768 = 590K
  - FFN: (768 × 3072 + 3072) + (3072 × 768 + 768) = 4.72M
  - Layer norms: 2 × (768 + 768) = 3K
  - **Итого на слой: ~7.1M**
- 6 слоёв: 6 × 7.1M = 42.6M
- **Всего: ~66M параметров** (91.8M в эмбеддингах, но они делятся с LM-головой)

### FFN: почему 3072?

Feed-Forward Network расширяет вектор в 4 раза (768 → 3072), потом сжимает обратно
(3072 → 768). Зачем?

Self-attention перемешивает информацию **между** токенами, но обрабатывает каждый
токен одинаково. FFN — это "мозг" каждого токена: в расширенном пространстве 3072
модель может выучить более сложные трансформации, чем в 768.

Исследования показали, что значительная часть "знаний" модели (факты о мире)
хранится именно в весах FFN, а не в attention.
